# Анализ раздетаченных (Detached) компонентов

Одна из главных целей проекта — поиск "самопальных" копий компонентов, которые не связаны с основной библиотекой.

## Как мы определяем статус элемента
В Figma API есть три состояния, которые нас интересуют:
1. **Official Instance**: Нода типа `INSTANCE`. У нее есть `componentId`. Тут всё прозрачно.
2. **Master Component**: Нода типа `COMPONENT` или `COMPONENT_SET`. Первоисточник.
3. **Detached Candidate**: Нода типа `FRAME` или `GROUP`, которая "притворяется" компонентом.

## Алгоритм обнаружения детачей (Heuristics)
Так как прямого флага "я был инстансом" в API нет, используем весовые коэффициенты:
- **Сходство структуры (40%)**: Сравнение имен и вложенности дочерних слоев. Если внутри "Frame" лежат слои "._Icon", ".Label", "._Background" — это 99% бывшая кнопка.
- **Сходство стилей (30%)**: Проверка уникальных комбинаций `fill`, `stroke`, `effect` и `text` стилей.
- **Совпадение имен (20%)**: Если фрейм называется так же, как мастер-компонент в библиотеке.
- **Геометрия (10%)**: Совпадение размеров и отступов (Auto-layout).

## Индексация внутренних слоев
Чтобы поиск "Найти все кнопки с иконкой поиска" работал, мы должны:
1. **Декомпозиция**: При парсинге инстанса записывать не только его имя, но и плоский список имен всех его дочерних слоев.
2. **Поле `layer_names`**: В БД (или ElasticSearch) хранить массив строк всех вложенных элементов.
3. **Теневой поиск**: При запросе "Header" искать не только по `name` компонента, но и по вхождениям в его структуру.

## Риски ложных срабатываний
- Простые контейнеры (стеки, сетки) могут быть похожи друг на друга.
- **Решение**: Игнорировать мелкие элементы (менее 3 слоев) при анализе на Detach.
