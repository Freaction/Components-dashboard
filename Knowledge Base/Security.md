# Безопасность и управление доступом

Защита интеллектуальной собственности (дизайна) и управление токенами Figma API.

## 1. Типы токенов и доступы
Для работы системы требуется один из двух типов аутентификации:

### Personal Access Token (PAT)
- **Для чего**: Быстрый старт, локальная разработка, внутренние инструменты.
- **Как получить**: Settings -> Account -> Personal access tokens.
- **Область доступа (Scopes)**: Достаточно прав `file_read`.
- **Риск**: Токен привязан к конкретному аккаунту. Если пользователь уволится, доступ пропадет.

### OAuth2 (Рекомендуется для Production)
- **Для чего**: Долгосрочное решение, не зависящее от конкретного пользователя.
- **Преимущество**: Позволяет обновлять доступ через `refresh_token`.

---

## 2. Безопасное хранение ключей
**Категорически запрещено** коммитить ключи в репозиторий.

### Локальная разработка
1. Создать файл `.env` в корне проекта (добавлен в `.gitignore`).
2. Добавить ключ: `FIGMA_ACCESS_TOKEN=figd_XXXXXXXXXXXX`.

### Серверная среда (Production)
- Использование систем управления секретами: **HashiCorp Vault**, **AWS Secrets Manager** или секреты в CI/CD (GitHub Secrets).
- Доступ к токенам должен быть только у ETL-воркеров.

---

## 3. Процедура обновления и ротации
1. **Плановая ротация**: Рекомендуется обновлять PAT раз в 3-6 месяцев.
2. **Экстренный отзыв**: Если токен попал в логи или был скомпрометирован:
   - Немедленно удалить токен в панели управления Figma.
   - Сгенерировать новый и обновить переменную окружения в Config Map / Vault.
   - Перезапустить ETL-сервисы.

---

## 4. Сетевая безопасность
- **Dashboard**: Доступ только через корпоративный VPN или SSO (Okta/Google).
- **База данных**: Закрыта для внешних подключений, доступ только из внутреннего контура.
- **Маскирование**: В логах системы токены должны заменяться на `figd_***`.
