# Глобальный поиск и аналитика компонентов (Global Search & Analytics)

## Цель
Позволить дизайн-системам и лидам фичевых команд быстро собирать скоуп реального использования компонентов (например, какие варианты `Badge` реально используются в макетах команд). Это нужно для актуализации UI Kit и сбора требований.

## Архитектура UI
Новый раздел приложения (отдельная вкладка `Search` или `Analytics`):
1. **Панель фильтров:**
   - Полнотекстовый поиск (по имени узла).
   - Фильтр по типам: `INSTANCE`, `COMPONENT`, `FRAME`, `TEXT` и т.д.
   - Фильтр по командам (Team) и файлам (File).
   - *Будущее:* Фильтр по свойствам компонента (например, `Variant = Primary`).
2. **Дерево результатов (Сгруппированное):**
   - Плоские списки на 1000+ элементов не читаются. Результаты должны группироваться:
     - 🔽 **Команда (Team Name)** (Найдено: 150)
       - 🔽 **Файл (File Name)** (Найдено: 150)
         - 🔹 Instance: Badge (Props: Size=Small, State=Success)
3. **Карточка деталей:**
   - Полные метаданные выбранного узла, ссылка "Open in Figma", история изменений.

## Необходимые доработки БД (DB Upgrades)
Текущая БД SQLite отлично справится, FTS5 индекс уже настроен. Но нам нужны расширения схемы:

1. **Сбор ВСЕХ свойств из Figma:**
   В таблицу `nodes` нужно добавить новую колонку `properties_json TEXT`. В нее сканер будет записывать `componentProperties` (для компонентов и инстансов), а также `layoutMode`, `fills` и прочие полезные данные, которые отдает Figma.
   *Миграция:* `ALTER TABLE nodes ADD COLUMN properties_json TEXT;`

2. **Эталонные файлы (UI Kit / Source of Truth):**
   В таблицу `team_files` нужно добавить флаг `is_reference BOOLEAN DEFAULT 0`.
   Это позволит нам на следующем этапе выявлять **detach-и** (отвязанные компоненты) или использование устаревших версий. Сценарий: мы сканируем "Reference" файл, берем из него эталонные компоненты, а затем сверяем с инстансами в файлах фичевых команд.

## Стратегия работы с сессиями
Мы сохраняем историю сканирований для обеспечения аналитики во времени (Time Machine):

1. **Кнопка "Scan All Teams":** 
   Вместо ручного сканирования каждой команды, на дашборде будет доступна глобальная кнопка, запускающая параллельное/последовательное сканирование всех актуальных файлов во всех командах. Каждое сканирование создает новый исторический срез.
2. **Поиск по последней сессии и сортировка:**
   В БД хранятся все исторические сессии (для будущих графиков и отчетов). Однако глобальный поиск по умолчанию фильтрует данные и ищет ТОЛЬКО по `status IN ('completed', 'failed', 'proceed')` и только по **самой последней сессии** для каждой команды/файла.
   *   **Сортировка по актуальности (Relevance):** Результаты по умолчанию сортируются по дате изменения файла Figma (`last_modified`). Это позволяет видеть в топе компоненты из тех файлов, над которыми дизайнеры работали недавно.
   *   **Отображение дат:** В интерфейсе отображается относительное время изменения файла (например, "2d ago", "1mo ago").

## Оптимизация производительности (Scale to Millions)
Для обеспечения мгновенного поиска на миллионах записей применены следующие техники:

1. **Денормализация (Page Name):**
   Имя страницы (`page_name`), на которой находится нода, сохраняется непосредственно в таблицу `nodes` в момент сканирования. Это исключает необходимость в тяжелых рекурсивных запросах (Recursive CTE) при каждом поиске.
2. **Индексация:**
   - Индекс на `last_modified` для быстрой сортировки по актуальности.
   - Индекс на `page_name` и `type` для фильтрации.
3. **Полнотекстовый поиск (FTS5):**
   Использование виртуальных таблиц SQLite FTS5 для поиска по именам и текстовому контенту, что обеспечивает скорость поиска за миллисекунды.
4. **Batch Insertion:**
   Запись данных в БД происходит пачками (по 100 нод), что снижает нагрузку на диск и ускоряет процесс индексации.
