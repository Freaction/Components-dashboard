# Управление командами и жизненный цикл данных

Раздел описывает логику администрирования команд, проектов и удаления данных.

## Реестр команд
Для мониторинга и управления необходимо хранить полный список подключенных команд.
- **Интерфейс**: Раздел в Admin UI со списком всех `team_id`, их названий и статусом синхронизации.
- **Статистика**: Кол-во проектов, файлов, инстансов и дата последнего обновления.

## Обработка изменений в Figma
1. **Смена названия команды/проекта**:
    - При каждом цикле Discovery система сравнивает `name` для существующих `id`.
    - При несовпадении — обновляет `name` в БД. ID (team_id, project_id) остаются константными.
2. **Перенос проекта (Transfer)**:
    - Если проект перемещен в другую команду, его `team_id` в нашей базе должен обновиться.
    - Если проект исчез из списка доступных (удален или отозваны права) — помечаем его как `orphaned` или удаляем.

## Удаление данных (Cleanup)
Должна быть предусмотрена возможность "забыть" данные:
- **Удаление команды**: Каскадное удаление всех проектов, файлов и нод, связанных с `team_id`.
- **Удаление проекта**: Удаление всех файлов и их индексов в ElasticSearch для конкретного `project_id`.
- **Soft vs Hard Delete**: 
    - Сначала помечаем как `deleted` (скрываем из поиска).
    - Через N дней — полное физическое удаление для очистки места.

## Синхронизация реестра
- Раз в сутки запускать полную сверку (Full Sync) для выявления удаленных в Figma элементов, которые не прислали вебхук.
